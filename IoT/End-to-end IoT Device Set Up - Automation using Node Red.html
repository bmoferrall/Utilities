Introduction

This page describes the use of node-red to automate end-to-end device set up as described on this Watson IoT page.

The current version of the flows can be viewed here.

Steps

Briefly, the steps to complete device set up are as follows:

 

  1. Create Device Type

  2. Create Device Instance

  3. Define a Schema for events

  4. Create a draft Event Type using the Schema

  5. Create a Physical Interface

  6. Add the Event Type to the Physical Interface

  7. Connect the Device Type to the Physical Interface

  8. Create a Schema for a Logical Interface

  9. Create a draft Logical Interface using the Schema

10. Add the Logical Interface to the Device Type

11. Define mappings between the Event and Logical Interface schema

12. Validate and Activate the Configuration

13. Publish an Event conforming to the schema in step 4

 

A more detailed description of the steps is documented here.

The APIs used in the above steps are described here.

The current version of the Event schema is attached here. The current version of the Logical Interface schema is attached here.

The current versions of the above Schemas are simplified to use all String properties. This will likely evolve to use different data types (such as dates).

Mappings for the Logical Interface

IoT uses the Jsonata language (link) to implement mappings between input events and an output logical interface. The logical interface is used to describe the tables (names, columns, data types) created in DB2. IoT creates the tables automatically and uses the mapping definitions to transform and route the input raw event data to DB2 (link).

This presents the possibility of implementing a data source/receiver framework similar to what is implemented in IOC. The mapping language and logical interfaces should make it possible, for example, to implement minimum/key/extended properties.

As an example, something like the following mapping might generate a LASTUPDATE field (using the built-in now() function) if the field is not present in the input event's payload:

"lastupdate": "$event.d.lastupdate?$event.d.lastupdate:$now()"

The following mapping will do the same, this time applying a date format to the timestamp (this feature would require an extension to Jsonata):

"lastupdate": "$event.d.lastupdate?$event.d.lastupdate:$moment().format('YYYY-MM-DD hh:mm:ss.000')"

Jsonata could be used to generate fields derived from other fields; it could also be used to build derived values based on aggregation over an interval such as a day.

The flow operates broadly as follows:

 

1. Load the current version of the configuration file

2. Each step in in the flow is traversed in sequence, as shown earlier.

3. If a step succeeds the flow proceeds to the next step. If a step fails the flow aborts and writes the relevant error message to an error log.

4. Some steps capture IDs from service responses and update the relevant element in the configuration file. If all steps succeed the configuration file is updated with newly captured IDs as a last step.

5. The full Set Up process can be reversed by running the Tear Down flow in full.

6. If the Set Up process fails prematurely the successful steps can be reversed by partially running the Tear Down flow from the point of failure. Alternatively the error can be corrected and the flow continued from the fail point.

Update Mappings between Event and Logical Interface

The small subflow below can be used to update the mappings. The mapping body might be implemented as follows:

 

var eventId = global.get("eventId");

var mappings = {

    "note": "$event.d.note?$event.d.note:''",
    "addresses": "$event.d.addresses?$event.d.addresses:''",
    "expires": "$event.d.expires?$event.d.expires:$now()",
    "msgType": "$event.d.msgType?$event.d.msgType:''",
    "code": "$event.d.code?$event.d.code:''",
    "areaDes": "$event.d.areaDes?$event.d.areaDes:''",
    "references": "$event.d.references?$event.d.references:''",
    "description": "$event.d.description?$event.d.description:''",
    "source": "$event.d.source?$event.d.source:''",
    "startdate": "$event.d.startdate?$event.d.startdate:$now()",
    "effective": "$event.d.effective?$event.d.effective:$now()",
    "senderName": "$event.d.senderName?$event.d.senderName:''",
    "urgency": "$event.d.urgency?$event.d.urgency:''",
    "web": "$event.d.web?$event.d.web:''",
    "contact": "$event.d.contact?$event.d.contact:''",
    "scope": "$event.d.scope?$event.d.scope:''",
    "incidents": "$event.d.incidents?$event.d.incidents:''",
    "event": "$event.d.event?$event.d.event:''",
    "headline": "$event.d.headline?$event.d.headline:''",
    "area": "$event.d.area?$event.d.area:''",
    "severity": "$event.d.severity?$event.d.severity:''",
    "identifier": "$event.d.identifier?$event.d.identifier:''",
    "resource": "$event.d.resource?$event.d.resource:''",
    "certainty": "$event.d.certainty?$event.d.certainty:''",
    "onset": "$event.d.onset?$event.d.onset:$now()",
    "sent": "$event.d.sent?$event.d.sent:$now()",
    "eventCode": "$event.d.eventCode?$event.d.eventCode:''",
    "enddate": "$event.d.enddate?$event.d.enddate:$now()",
    "sender": "$event.d.sender?$event.d.sender:''",
    "instruction": "$event.d.instruction?$event.d.instruction:''",
    "geocode": "$event.d.geocode?$event.d.geocode:''",
    "restriction": "$event.d.restriction?$event.d.restriction:''",
    "location": "$event.d.location?$event.d.location:''",
    "lastupdate": "$event.d.lastupdate?$event.d.lastupdate:$now()",
    "category": "$event.d.category?$event.d.category:''",
    "parameters": "$event.d.parameters?$event.d.parameters:''",
    "status": "$event.d.status?$event.d.status:''"

};

var eventMappings = {};
eventMappings[eventId] = mappings;

msg.payload = {
    "logicalInterfaceId": global.get("logicalInterfaceId"),
    "notificationStrategy": "on-every-event",
    "propertyMappings": eventMappings,
    "version": "draft",
    "updated": now()
};
msg.headers = {"Content-Type":"application/json"};

return msg;
