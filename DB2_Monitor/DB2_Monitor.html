DB2 Monitoring Scripts for Linux

About

Database servers have proven to be a major source of bottlenecks in many of our applications.

Disk I/O issues and excessive CPU utilization can be caused, for example, by poorly designed data models, inefficient SQL queries, lack of appropriate indexing, suboptimal database configuration settings and lack of database maintenance.

DB2 exposes a number of monitoring interfaces (e.g., MON_GET_PKG_CACHE_STMT, MON_GET_TABLE, MON_GET_INDEX, MON_GET_CONNECTION) that allow us to query what is happening in our databases at a very fine-grained level. Some of the metrics we can access using these interfaces include:

  • how much CPU time a query requires for execution
  • how much time a connection agent has to wait before it can execute a statement
  • how many times a query has been executed over a period of time
  • the isolation level that is used by a statement
  • how many rows a query has to read from tables or indexes
  • how many sort operations are required in a query execution

We can create formulas based on a combination of metrics, and take differences over a period of time, to derive further useful metrics, for example:

  • how efficiently we are using buffer pools (buffer pool hit ratios)
  • how many SQL statements are executed per second (an indication of SQL throughput)
  • how many direct reads to disk are occurring per second (due to non-inlined Large Object columns)
  • how many rows are being read per second
  • which tables and indexes are most often accessed during a test phase
  • which tables are undergoing frequent expensive scans
  • the extent of lock and sort activity.

We have written a number of SQL scripts that query these interfaces and generate reports. A driver shell script will invoke these SQL scripts and allow us to automate the process of generating these reports during test runs. If the reports indicate that one or more SQL statements are performing poorly we can use additional tooling to explore the possibility of tuning these statements: by rewriting the queries, adding indexes, creating intermediate summary tables (MQTs), etc. Data Studio has a very useful tool called ‘Optim Access Plan Explorer’, which graphically displays the execution plan for a statement and the estimated relative cost of executing each part of the plan. You can also install a plugin called ‘Optim Query Workload Tuner’ (at the time of writing, part numbers CIK8GML and CIK8IML in Xtreme Leverage for the Windows and Linux versions respectively) which will give recommendations on how you might rewrite a query and whether an index or indexes can help improve the performance of that query.

Installing the Tool

Prerequisites

1. sysstat package is installed. Check using 'yum list installed sysstat'
   The package can be installed from here: http://sebastien.godard.pagesperso-orange.fr/download.html
    
     Once installed, increase the frequency of background stats collection from the default 10 min to 1 min
     Edit the file '/etc/cron.d/sysstat' and configure sa1 as follows
    
     * * * * * root /usr/lib64/sa/sa1 1 1


2. Turn on DB monitor switches for each instance to be monitored.
   If these switches are not enabled some of the report statistics will be zero
   You can check if they have been set with 'db2 get dbm cfg | grep dft_mon'
   To enable:
      db attach to <instance_name>
      db2 update dbm cfg using dft_mon_bufpool ON
      db2 update dbm cfg using dft_mon_lock ON
      db2 update dbm cfg using dft_mon_stmt ON
      db2 update dbm cfg using dft_mon_table ON
      db2 update dbm cfg using dft_mon_timestamp ON
      db2 update dbm cfg using dft_mon_uow ON
      db2 update dbm cfg using dft_mon_sort ON
   Restart the instance:
      db2stop force
      db2start


 3. (optional) Install DB2's explain facility in the databases to be monitored.
   If installed, the utility will export Explain Plans for the top 15 slowest-executing statements
   Example for DB2 11.1:
   db2 connect to <dbname>
   db2 -ntf /opt/ibm/db2/V11.1/misc/EXPLAIN.DDL
   db2 connect reset


4. (optional) Install the 'unix2dos' utility if you plan to view the output in Windows
   yum install unix2dos

Setup

1. Create script directory ($scriptDir) and copy the sh and sql scripts into it.
2. Execute the following commands on the contents of the script directory:
   dos2unix $scriptDir/*
   chmod -R 755 $scriptDir
   chmod +x $scriptDir/*.sh

Description

   The DB monitor tool uses a diff approach to capture SQL activity during the monitor period only
   If the load  you are monitoring is constant over the duration of a test you can simply start monitoring before the test
   and set the monitor duration to the duration of the test, or slightly longer

   If varying loads are applied in phases you can configure the tool to match those phases.
   Output files will be generated for each monitor phase (_1, _2, _3, etc)
   For example, given the following load scenario:

   | ramp up to 10 users over 1 min | hold at 10 users for 20 mins |
          ...       | ramp up to 20 users over 1 min | hold at 20 users for 20 mins |

   You can configure the DB monitoring to synchronize with the hold phases as follows:

   dbmon.sh -s 1200 -i 60 -n 2 -c <instance> -u <user> -p <password> -d 'DB1,DB2'

   Output files will be saved to /var/log/db/mon/XX/DBLIST/, where XX is the 2-digit day number
   For example: /var/log/db/mon/24/DB1_DB2/DB1_DB2_0820_0958.zip

---------------

Note that the script should be run as root. It will in turn invoke the script 'dbmon_report.sh' as the specified (local) database user. The list of databases to be monitored should be in quotes and separated by commas with no spaces (as in the example above). The script will also execute iostat, mpstat (from the sysstat suite) and top for the duration of the test. A log of the script execution will be placed in the output directory.

If you are monitoring catalogued databases from a remote server that has DB2 client tools installed, iostat/mpstat/top data will be for the client machine, so can be ignored.

Executing Individual Scripts

Some of the individual sql scripts can usefully be run on their own. For example:

1. View stats for currently executing queries

db2 connect to <DBNAME>

db2 -ntf getmoncurrentsql.sql > current_<dbname>.txt

db2 connect reset

 

This is useful to monitor slow-running or potentially deadlocked queries.

2. View Connection Details

db2 connect to <DBNAME>

db2 -ntf getdbconnectionsinfo.sql > connections_<dbname>.txt

db2 connect reset

3. View stats for recently completed queries

db2 connect to <DBNAME>

db2 -ntf getdbsqlcache.sql > cache_<dbname>.txt

db2 connect reset

 

From the output of this script you can get information such as the time when the query was first compiled, how often a query has been executed, various execution component times, rows read/returned (ideally should be <10), average rows read per execution, deadlocks, physical table/index disk reads, statement text, etc. More information about the fields included in the output can be found here (some of the field names in the output will have been renamed, or used to derive new fields, so refer to getdbsqlcache.sql when correlating). The key timing fields are as follows:

 

AVG_CPU_USEC:         time in microseconds to execute a statement (not including any wait time)
AVG_ACT_MSEC:          total time in milliseconds to execute statement, including processor queue time/IO wait time/Lock wait time
AVGACT_WT_MSEC:    additional overhead due to IO wait time and/or lock wait time (included in AVG_ACT_MSEC value)
AVGLOCK_WT_MSEC: overhead due to lock wait time (included in AVG_ACT_MSEC value).
ASTMT_EXEC_MSEC:  this value includes any additional time due to stored procedure calls in the statement (AROUTINE_TIME_MSECS).

 

If  you want to measure the rate at which a query or queries are being executed flush the package cache first:

 

db2 connect to <DBNAME>

db2 flush package cache dynamic

***Run tests for known period of time***

db2 -ntf getdbsqlcache.sql > cache_<dbname>.txt

db2 connect reset

 

Now, the EXECS time and the test duration can be used to work out average query rates. If you know that particular queries are caused by a specific service you can run flush the cache, execute the service, then capture the stats to see what queries are executed. Sometimes a flush doesn't eject all entries from the cache, so you should confirm when the entry was added to the cache from the 'INSERT_TIMESTAMP' column.

DB2 Maintenance Script for Linux

Note that this script is not intended as a substitute for DB2's maintenance scheduler. However, you may find it useful under certain circumstances.

This script will run maintenance on the specified schemas in the specified database. It's recommended to run a script like this on a daily basis, and after large data loads, to maintain the health of a database and consistent performance.

  1. It executes REORGCHK to check which tables in the specified schemas require reorganization
  2. For those tables indicated as requiring a REORG it checks the size of the table. If the size is below a (configurable) threshold, the table is reorganized. Otherwise, the REORG is skipped.
  3. It executes RUNSTATS on all tables in the specified schema with basic and distribution stats collected for all columns and extended index stats. It also sets a profile so that auto runstats uses the same options
  4. Performs a full rebind of all packages in the specified database.

>>>>>

Note: step 4 (db2rbind) can hang on occasion. When that happens write access to tables may be blocked and applications may stop working. To kill the hanging application first run 'db2 list applications' as below to identify the application ID (in this case 15288):

 

 db2 list applications | grep db2rbind

DB2I1OWN db2rbind       15288      *LOCAL.db2i1own.180529004227                                   IOCDB    1

 

Then, run:

 

db2 "force application (15288)"

 

To reduce the risk of the rebind command hanging close application connections (e.g., shut down Liberty) before running the maintenance script.

<<<<<<

The script could be scheduled to run on a regular basis using a cron task.

As an alternative to this approach you can  schedule an offline maintenance period in DB2. The disadvantage is that you can't REBIND packages using DB2's auto maintenance, so this would have to be scheduled manually to happen after REORG/RUNSTATS have completed.

If packages are used extensively in a database it's essential to rebind them on a regular basis to take advantage of updated statistics. By default, packages are bound with the 'REOPT NONE' option, which means that access plans are hard-coded during initial compilation of the package. As data is added to a DB the access plan will become increasingly suboptimal and degrade query performance.

The script can be downloaded from here. Make script executable and run dos2unix:
 

dos2unix db_maint.sh
chmod 777 db_maint.sh
 

USAGE: ./db_maint.sh -d DBNAME -s 'SCHEMA1 SCHEMA2...' -r REORGTHRESH
Required:
  -d: database name
  -s: space-separated list of schema names
Optional:
  -r: reorg threshold in MB (only reorg tables below this size, default 100MB)

It's good practice to include the SYSIBM schema when doing maintenance as system tables can become significantly de-fragmented over time.

 

Example (run as db2i1own):

./db_maint.sh -d WIHDB -s 'WIH SYSIBM'

Example (run as root):

su - db2i1own -c "/<path_to_db_maint_script>/db_maint.sh -d WIHDB -s 'WIH CFG VDS EAM SYSIBM'"

If setting up a cron task configure it as root user as follows (this example runs it every day at 2am):

0 2 * * * su - db2i1own -c "/<path_to_db_maint_script>/db_maint.sh -d WIHDB -s 'WIH SYSIBM'"
